# Build an Advanced Hook Stack

This tutorial is for teams that already understand [Your First Hook](/tutorials/first-hook), [The Six-Step Loop](/tutorials/six-step-loop), and [Project Setup](/configuration).

It takes a Codex-style “workflow enforcement” setup and translates it into **Claude Code flavor** without making the solution Python-specific.

## 1. Translate the Codex setup before you copy it

The Codex version usually looks like this:

- one global instructions file
- a few hook scripts
- one config file that wires the hooks together
- a validation checklist and smoke test

In Claude Code, the closest equivalents are:

| Codex concept | Claude Code equivalent |
| --- | --- |
| `~/.codex/instructions/global-model-instructions.md` | shared workflow rules in `CLAUDE.md`, plus optional scoped `.claude/rules/*.md` files |
| `~/.codex/config.toml` hook wiring | `.claude/settings.json` or `~/.claude/settings.json` |
| `apply_patch` guard | `Write|Edit|MultiEdit` hook matcher |
| `exec_command` guard | `Bash` hook matcher |
| one post-tool audit point | `PostToolUse` **and** `PostToolUseFailure` in Claude Code |

That last row matters.

If you want a RED signal from a **failing** test command, Claude Code needs `PostToolUseFailure` because failed tool calls do not flow through ordinary `PostToolUse`.

## 2. Put the workflow rules in the right Claude place

Codex has a direct `model_instructions_file` setting.

Claude Code does **not** use that same mechanism.

The practical Claude translation is:

1. put the shared cross-repo workflow rules in `CLAUDE.md`
2. keep narrower path-specific rules in `.claude/rules/*.md`
3. use hooks only for the parts that are **machine-checkable**

Start with a `CLAUDE.md` addendum like the downloadable `CLAUDE.advanced-hooks.md` file:

```md
# Advanced Hook Workflow Addendum

- Always read applicable AGENTS.md before substantive work.
- Use max TDD: name the tests first, then write or update tests before implementation.
- Do not edit non-test source without a recent RED signal.
- Reproduce the exact user command before claiming a fix when feasible.
- For CLI or output bugs, add at least one integration test for the real entrypoint.
- If the user gives an exact error string, reproduce that exact string in a test when feasible.
- For stdout or stderr requirements, verify startup output, progress output, final output, and flushing.
- Do not let unrelated failing tests redefine the task.
- No silent failures; surface progress and errors clearly.
```

Keep the judgment-heavy rules in `CLAUDE.md`.

Use hooks for the narrow parts you can actually prove:

- source edits need RED first
- test commands can create RED or GREEN signals
- the session cannot stop after source edits unless GREEN happened later

## 3. Install the advanced hook files

The advanced pack includes these files:

- `CLAUDE.advanced-hooks.md`
- `.claude/settings.advanced-hooks.json`
- `.claude/hooks/hooklib.sh`
- `.claude/hooks/session-start.sh`
- `.claude/hooks/pre-tool-use-guard.sh`
- `.claude/hooks/post-tool-use-audit.sh`
- `.claude/hooks/stop-guard.sh`

The install flow is:

1. merge `CLAUDE.advanced-hooks.md` into your repo `CLAUDE.md`
2. merge the hook entries from `settings.advanced-hooks.json` into `.claude/settings.json`
3. copy the shell hook files into `.claude/hooks/`
4. make sure `.claude/state/` exists

These shell scripts use `jq` for JSON parsing and state updates.
If your team prefers another runtime, port the same logic, but keep the behavior the same.

## 4. Use the Claude-flavored settings shape

This is the key part of the downloadable settings file:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/session-start.sh",
            "timeout": 10
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/pre-tool-use-guard.sh",
            "timeout": 10
          }
        ]
      },
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/pre-tool-use-guard.sh",
            "timeout": 10
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit|Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/post-tool-use-audit.sh",
            "timeout": 10
          }
        ]
      }
    ],
    "PostToolUseFailure": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/post-tool-use-audit.sh",
            "timeout": 10
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/stop-guard.sh",
            "timeout": 10
          }
        ]
      }
    ]
  }
}
```

### Why this differs from the Codex version

- `Write|Edit|MultiEdit` is the Claude-side equivalent of guarding source edits
- `Bash` is the Claude-side equivalent of guarding command execution
- `PostToolUseFailure` is added because failing test commands should still update hook state
- the `Stop` hook is a **command** guard here, not an agent hook, because the goal is a simple machine-checkable GREEN gate

If you want Claude itself to reason about whether the **exact user command** was rerun, upgrade the `Stop` hook later to an `agent` hook. The downloadable pack keeps that part intentionally simple.

## 5. Understand what each hook file does

### `hooklib.sh`

Shared helpers:

- read hook JSON from stdin
- load and save `.claude/state/advanced-hook-state.json`
- classify source files versus docs, tests, or config
- recognize likely test commands

### `session-start.sh`

This initializes the state file at session start so the other hooks have somewhere to record RED, GREEN, and edit history.

### `pre-tool-use-guard.sh`

This is the hard gate.

It blocks:

- source edits when there is no recent RED signal
- obviously destructive shell commands

It allows:

- tests, docs, config, and infrastructure edits
- source edits after a recent failing test run

### `post-tool-use-audit.sh`

This is the state recorder.

It tracks:

- the most recent source edit
- the most recent failing test run as RED
- the most recent passing test run as GREEN

Wire it to both `PostToolUse` and `PostToolUseFailure`.

That is the Claude-specific trick that makes this translation work.

### `stop-guard.sh`

This blocks task completion when:

- source files were edited
- and no later passing test run was recorded

That gives you a narrow mechanical SHIP gate without trying to encode every line of `CLAUDE.md` into automation.

## 6. Validate the setup

Run the same validation sequence every time you install or modify this stack:

```bash
claude --version
bash -n .claude/hooks/hooklib.sh
bash -n .claude/hooks/session-start.sh
bash -n .claude/hooks/pre-tool-use-guard.sh
bash -n .claude/hooks/post-tool-use-audit.sh
bash -n .claude/hooks/stop-guard.sh
```

## 7. Smoke test the hook behavior

Use this exact sequence:

### A. Confirm source edit is blocked without RED

Prompt:

```text
Edit a non-test source file.
```

**What you should see:** Claude is blocked and told to create a failing test first.

### B. Confirm a failing test creates RED

Prompt:

```text
Write the smallest failing test for the change and run it.
```

**What you should see:** the test command fails, but `PostToolUseFailure` records a RED signal instead of losing that information.

### C. Confirm source edit is now allowed

Prompt:

```text
Now implement the source change.
```

**What you should see:** the edit goes through because RED exists.

### D. Confirm Stop blocks without GREEN

Prompt:

```text
We are done. Finish the task.
```

**What you should see:** the Stop guard blocks completion because a source edit happened after RED and no passing test has been recorded yet.

### E. Confirm Stop allows completion after GREEN

Prompt:

```text
Run the relevant tests until they pass, then finish the task.
```

**What you should see:** a successful test command records GREEN and the Stop guard allows completion.

## 8. Keep the scope disciplined

Do **not** use hooks for everything from the Codex rule file.

Keep these in `CLAUDE.md` or review habits instead:

- naming the tests first
- deciding which test is the right one
- determining the closest real runtime entrypoint
- deciding whether a specific bug truly needs integration coverage

The hook stack should enforce the sharp edge, not replace judgment.
